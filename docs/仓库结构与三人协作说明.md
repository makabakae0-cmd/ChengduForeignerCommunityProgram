# 仓库结构与三人协作说明

## 1. 文档目的

这份文档面向当前 3 位开发人员和后续接手者，目的不是重复产品需求，而是把**仓库怎么分层、每个目录做什么、每个人负责什么、怎样按统一流程开发和联调**讲清楚。

当前项目已经不是“纯空目录”状态，而是一个可运行的 monorepo 骨架：

- `apps/mobile`：uni-app 小程序前台
- `apps/admin`：Vue 轻后台
- `apps/api`：统一 BFF / CloudBase 接口层
- `packages/shared`：三端共享契约层
- `docs`：文档层

同时，仓库里也已经存在一些**构建产物目录**和**依赖目录**。这些目录不是业务开发入口，不能和源码目录混着理解。

---

## 2. 先看整体：哪些目录能改，哪些目录不要改

### 2.1 业务源码主目录

这些目录是日常开发真正要进入的地方：

- `apps/mobile`
- `apps/admin`
- `apps/api`
- `packages/shared`
- `docs`

### 2.2 生成目录 / 依赖目录

这些目录不是业务源码目录，不应手工修改：

- `node_modules/`
- `apps/mobile/node_modules/`
- `apps/admin/node_modules/`
- `apps/api/node_modules/`
- `.pnpm-store/`
- `apps/mobile/dist/`
- `apps/admin/dist/`

这些目录的含义：

- `node_modules/` 和各子项目里的 `node_modules/`：依赖安装结果
- `.pnpm-store/`：pnpm 的包缓存
- `dist/`：构建输出结果

规则：

- 不要往这些目录里提交业务代码
- 不要手改这些目录里的文件来“临时修问题”
- 发现构建结果不对，要改源码，不要改 `dist`

---

## 3. 根目录结构说明

根目录负责整个 monorepo 的工程管理，不负责某个业务模块本身。

### 3.1 `/apps`

作用：

- 存放应用层项目
- 每个子目录是一个可独立运行的应用

当前包含：

- `apps/mobile`
- `apps/admin`
- `apps/api`

这里应该放什么：

- 完整应用项目
- 各自的入口、页面、路由、应用内组件、应用层 API 消费逻辑

这里不应该放什么：

- 三端共享的 DTO / schema / enum
- 跨应用复用的接口契约定义

共享内容必须放到 `packages/shared`。

### 3.2 `/packages`

作用：

- 存放共享包
- 当前项目里最核心的是 `packages/shared`

这里应该放什么：

- 三端共用的接口契约
- 共享类型
- 共享校验规则
- mock 数据和通用 client

这里不应该放什么：

- 某个页面私有的展示逻辑
- 某个应用独有的 UI 组件

### 3.3 `/docs`

作用：

- 放项目文档
- 放共享约定、架构说明、协作说明、接口说明、部署说明

这里应该放什么：

- 产品设计文档
- 仓库说明
- 三人协作文档
- 环境配置和联调说明

这里不应该放什么：

- 临时截图堆积
- 未命名的随手记录

建议规则：

- 任何会影响多人协作的约定，先写进 `docs`
- 再去改代码

### 3.4 根目录配置文件

#### `package.json`

作用：

- 管理整个 monorepo 的顶层脚本
- 统一定义开发、类型检查、测试、lint 命令

当前重点脚本：

- `corepack pnpm --filter @community-map/mobile dev:h5`
- `corepack pnpm --filter @community-map/mobile dev:mp-weixin`
- `corepack pnpm --filter @community-map/admin dev`
- `corepack pnpm --filter @community-map/api dev`
- `corepack pnpm typecheck`
- `corepack pnpm test`

什么时候要改：

- 增加新的全局脚本
- 调整开发流程
- 引入新的顶层工具

谁可以改：

- 赵冉杰主责
- 其他人如需改动，必须先说明影响面

#### `pnpm-workspace.yaml`

作用：

- 告诉 pnpm 哪些目录属于 workspace

什么时候要改：

- 增加新的 app 或 package

默认不应该被模块前端同学改动。

#### `tsconfig.base.json` / `tsconfig.json`

作用：

- 定义整个仓库共用的 TypeScript 基线配置
- 统一路径映射和严格性

什么时候要改：

- 调整共享路径别名
- 调整 TS 严格度
- 增加全局编译规则

这类变更会影响三端，必须谨慎。

#### `eslint.config.mjs`

作用：

- 定义全仓库 lint 规则

什么时候要改：

- 团队统一代码风格变化
- 引入新文件类型或新校验规则

#### `vitest.config.ts`

作用：

- 定义测试扫描规则
- 统一测试运行环境

什么时候要改：

- 新增测试目录规范
- 修改测试运行方式

#### `.env.example`

作用：

- 给开发人员一个环境变量模板
- 说明项目需要哪些运行时配置

这里应该放什么：

- 环境变量名字
- 示例值或空值

这里不应该放什么：

- 真实密钥
- 生产环境配置

#### `.gitignore`

作用：

- 过滤依赖、构建产物、本地缓存和私有文件

不要为了“方便提交”去删掉关键忽略规则。

## 4. `apps/mobile`：小程序前台

`apps/mobile` 是 uni-app 前台项目，负责用户可见的小程序页面和基础前端状态。

小程序端 UI 默认采用 TDesign MiniProgram。新增页面、组件、表单、弹窗、按钮、列表、导航栏、反馈提示等，应优先使用或参考 TDesign MiniProgram；自定义样式应保持 TDesign MiniProgram 的视觉语言。

### 4.1 这个项目的定位

职责：

- 承载首页和三大模块前台页面
- 管理小程序端导航、页面结构、基础状态
- 消费统一 API client

不负责：

- 定义业务字段本身
- 直接决定后端返回结构
- 绕过 BFF 直接写核心业务数据

### 4.2 `apps/mobile/index.html`

作用：

- H5 构建入口

说明：

- 它服务于 `build:h5`
- 不是小程序业务逻辑文件

### 4.3 `apps/mobile/package.json`

作用：

- 管理 mobile 自己的依赖和脚本

当前重点：

- 使用 `uni-app + vite`
- 使用 TDesign MiniProgram 作为微信小程序端主要 UI 组件库
- 包含 `dev:h5`、`dev:mp-weixin`、`build:h5`、`build:mp-weixin`

这是移动端源码主目录。

#### `apps/mobile/src/App.vue`

作用：

- 应用级入口组件

应该放什么：

#### `apps/mobile/src/pages.json`

## 6. `apps/api`：统一 BFF / CloudBase 接口层

`apps/api` 是整个项目最关键的中间层。它决定 mobile 和 admin 最终怎么拿数据，也负责把 mock 模式和 CloudBase 模式统一起来。

### 6.1 这个项目的定位

职责：

- 提供统一 API
- 做参数校验
- 做鉴权和角色判断
- 统一成功 / 失败响应包络
- 决定走 mock 还是 cloudbase provider

不负责：

- 页面渲染
- 自己另起一套前端展示类型

### 6.2 `apps/api/src/app.ts`

作用：

- 本地 Koa 服务入口
- 用于本地开发和测试

当前已经做了什么：

- 注册错误中间件
- 注册 requestId
- 注册 body parser
- 注入 provider
- 注入 actorMiddleware
- 装配 announcements / auth / discover / events / files / notifications / places 路由

这意味着：

- 这里是“总装配层”
- 不能继续往里面堆模块业务逻辑

### 6.3 `apps/api/src/cloudbase.ts`

作用：

- CloudBase HTTP 云函数入口

意义：

- 这是部署到 CloudBase 的正式入口
- `app.ts` 是本地 Koa 入口
- `cloudbase.ts` 是云函数入口

两者应该保持接口行为一致。

### 6.4 `apps/api/src/dev.ts`

作用：

- 本地开发启动脚本入口

### 6.5 `apps/api/src/index.ts`

作用：

- 导出本地服务 callback

### 6.6 `apps/api/src/lib`

职责：

- 放 API 共用基础能力

当前已有：

- `auth.ts`
- `errors.ts`
- `http.ts`

这里应该放什么：

- 鉴权中间件
- 统一错误类
- 请求解析和响应包络

这里不应该放什么：

- 具体 events / posts / places 业务实现

### 6.7 `apps/api/src/routes`

职责：

- 放 HTTP 路由层

当前已有：

- `announcements.ts`
- `auth.ts`
- `discover.ts`
- `events.ts`
- `files.ts`
- `notifications.ts`
- `places.ts`

当前路线是“薄路由”：

- 路由只负责注册 URL
- 用 zod schema 校验参数
- 调 provider
- 统一返回结果

这里应该放什么：

- HTTP path 定义
- 入参校验
- 轻量权限调用

这里不应该放什么：

- 复杂查询逻辑
- 数据库查询细节
- 大量业务分支

### 6.8 `apps/api/src/providers`

职责：

- 放数据来源适配层

当前已有：

- `index.ts`
- `types.ts`
- `mock/`
- `cloudbase/`

#### `apps/api/src/providers/mock`

作用：

- 本地 mock 数据模式
- 让前端在后端未接真实云环境之前先联调

#### `apps/api/src/providers/cloudbase`

作用：

- 未来的 CloudBase 正式实现

当前状态：

- 已有目录和入口
- 具体业务实现还要继续补

这里以后应该放什么：

- 真实数据库读写
- 文件服务接入
- CloudBase SDK 封装

### 6.9 预留的业务目录

当前 `apps/api/src` 下面还有这些目录：

- `admin`
- `announcements`
- `auth`
- `events`
- `notifications`
- `places`
- `posts`
- `shared`

这些目录现在多数还是空目录或保留位。它们不是没用，而是**下一阶段业务实现真正应该落进去的目录**。

建议后续在这些目录里放：

- `service.ts`：业务服务逻辑
- `repository.ts`：数据访问封装
- `mapper.ts`：数据库对象与 DTO 转换
- `constants.ts`：模块内部常量

不要继续把所有实现直接塞在 `routes/` 里。正确方向是：

- `routes/` 负责 HTTP
- 业务目录负责模块内部实现

### 6.10 `apps/api/test`

职责：

- 放 API 测试

当前已有：

- `app.spec.ts`
- `cloudbase.spec.ts`

它的意义非常重要：

- 验证 mock 路由可跑
- 验证权限逻辑
- 验证本地入口和 cloudbase 入口包络一致

## 7. `packages/shared`：唯一共享契约源

这是整个仓库最重要的共享层。

### 7.1 它的定位

它解决的问题是：

- mobile / admin / api 三端不能各自理解接口
- 所有核心 DTO、schema、enum、path、client 必须有一个唯一来源

## 10. 统一开发流程：必须走 contract-first

这是保证项目最终能真正实现的核心流程。

### 10.1 正确顺序

1. 先在 `packages/shared` 确认 DTO、schema、enum、path  
2. 再在 `apps/api` 落 BFF 路由与 provider  
3. 再更新 `apps/mobile/src/api` 和 `apps/admin/src/api` 的消费方式  
4. 最后各自开发页面

### 10.2 为什么顺序不能反

如果前台先随便写字段：

- mobile 的字段会跟 admin 不一致
- admin 会逼后端临时兼容
- BFF 会被迫写脏逻辑
- 后面 shared 层失去约束力

所以禁止：

- 在页面里手写接口路径
- 页面自己重新定义核心 DTO
- 前台先写死字段，再要求后端适配

### 10.3 共享契约变更流程

当 Emma 或刘知行发现接口不够用时：

1. 先明确需要新增或调整的字段
2. 写清楚这个字段属于请求、响应还是页面局部 UI 数据
3. 若是共享字段，先改 `packages/shared`
4. 再由赵冉杰同步 `apps/api`
5. 再由使用方页面改接入

### 10.4 PR 合并顺序

推荐顺序：

1. 共享契约 PR 先合
2. 后端 / BFF PR 第二步合
3. 后台和各模块前台 PR 再分别合并

不能反过来。

## 12. 为什么现在这种分工和结构能最终落地

### 12.1 页面边界已经天然分开

前台三条主业务线已经拆到不同目录：

- `events`
- `discover`
- `places`

这让 Emma 和刘知行不会天然冲突，赵冉杰也有独立的 places 和集成职责。

### 12.2 共享层只有一个来源

`packages/shared` 统一了：

- contract
- schema
- enum
- type
- client
- mock

这能防止：

- mobile 一套字段
- admin 一套字段
- api 一套字段

### 12.3 BFF 已经按 route / provider 分层

当前 API 层不是一坨入口文件，而是：

- `routes` 负责 HTTP
- `lib` 负责基础能力
- `providers` 负责数据来源切换

这为后续继续把业务服务下沉到 `src/events`、`src/posts`、`src/places` 等目录留出了空间。

### 12.4 mock 与 cloudbase 双模式减少阻塞

现在的结构允许：

- 前台先用 mock 联调
- 后端后补 CloudBase 实现

这能避免“前端全体等待后端接完云环境”。

## 13. 最容易出问题的地方与规避办法

### 13.1 不要在页面里手写接口路径

错误做法：

- 在 `pages/*.vue` 直接写 `/events/...`

正确做法：

- 统一走 `apps/mobile/src/api/client.ts`
- 再由 shared client 和 shared paths 处理

### 13.2 不要绕过 `packages/shared` 自己定义重复类型

错误做法：

- 在 mobile 自己写 `type Event = ...`
- 在 admin 再写一份 `EventRow = ...`

正确做法：

- 核心 DTO 从 `packages/shared` 来
- 页面局部类型才放本地

### 13.3 不要把模块逻辑继续堆进总入口

不要把复杂逻辑直接塞进：

- `apps/api/src/app.ts`
- `apps/admin/src/router/index.ts`
- `apps/mobile/src/stores/app-store.ts`

原因：

- 这些文件是装配层，不是业务承载层

### 13.4 不要修改生成目录

禁止把修复做在以下目录：

- `dist/`
- `node_modules/`
- `.pnpm-store/`
